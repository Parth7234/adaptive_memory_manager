Based on the project structure and the README.md documentation, I will break down the samsung folder for you. This project implements a Context-Aware Adaptive Memory Management System designed to optimize application caching, pre-loading, and eviction on smartphones and edge devices using Machine Learning.

Here is a detailed explanation of each folder, its purpose, the data used, and the ML techniques applied.

📂 Folder & File Breakdown
1. src/ (Source Code)
This is the core directory containing all the logic for data generation, machine learning, and memory management.

data_generator.py: Creates synthetic datasets that simulate real-world smartphone usage. Since real app usage data is sensitive, this script builds realistic sequences for 50 users over 30 days. It defines user habits (like time-of-day biases and sequential app switches like Camera -> Gallery).
predictor.py: The "Brain" of the system. This file contains the Machine Learning models (LSTM and Markov Chains) used to predict what app a user will open next.
memory_manager.py: The "Brawn" of the system. It simulates device memory (RAM). It contains a baseline LRU (Least-Recently-Used) cache manager and an Adaptive ML-driven manager. It uses the predictions from predictor.py to smartly pre-load apps and evict unused apps. It also has specific logic for handling Generative AI Key-Value (KV) cache workloads.
visualize.py: Handles generating charts and graphs (like prediction accuracy and KPI dashboards) to compare how the standard memory manager performs against the ML-driven one.
2. data/ (Datasets)
Contains the data generated by data_generator.py that is used to train the ML models and run the simulation.

app_usage_logs.csv: Contains roughly 117,000 app usage records. Each record simulates a user opening an app, recording the user ID, app ID, and timestamp.
kv_cache_workload.csv: Contains simulated workloads specifically for Generative AI apps (like AI Assistants or Image Generators), detailing inference requests and context lengths.
app_metadata.json: Defines the properties of the 20 simulated applications (their names, memory footprint, type, etc.).
3. models/ (Trained Machine Learning Models)
Stores the saved parameters of the models after they finish training so they don't have to be retrained every time you run the system.

best_predictor.pth: The saved weights of the trained LSTM (Neural Network) model built using PyTorch.
markov_model.pkl: A saved Python object containing the transition probability matrices for the Markov Chain model.
training_history.json: A log of the metrics (loss, accuracy) recorded during the model's training phase.
4. results/ (Outputs & Benchmarks)
(Generated after running the code) Contains the benchmark comparisons between standard LRU memory management and the ML adaptive memory management, as well as generated visualizations (PNGs/JSONs) showing hit rates, load times, and page faults.

5. Root Level Files
run.py: The main orchestration script. Running this single file executes the entire pipeline: generating data → training models → running simulations → visualizing results.
simulate.py: The simulation engine that replays the app_usage_logs.csv trace through both the standard and ML-driven memory managers to benchmark their performance.
🧠 ML Techniques Used
The system uses a Hybrid Ensemble Approach, meaning it combines two different types of models to get the best of both worlds:

LSTM with Attention Mechanism (Deep Learning):
What it is: Long Short-Term Memory (LSTM) is a type of Recurrent Neural Network (RNN) excellent at understanding sequences and time-series data. The "Attention" mechanism helps the model focus on the most relevant past actions.
How it works here: It takes in User Embeddings (who the user is), App Embeddings (what apps they use), and Time Encodings (hour of day, day of week) to learn complex, generalized patterns across all users.
Markov Chains (Probabilistic Model):
What it is: A mathematical system that predicts the next state based entirely on the current and immediate previous state.
How it works here: It uses a 2nd-order Markov Chain to capture deterministic, per-user habits. If "User A" always opens "Maps" exactly after "Calendar", the Markov Chain learns this specific user habit perfectly with zero training time.
The Ensemble: The final prediction is a weighted combination (e.g., 30% LSTM + 70% Markov). The Markov chain handles immediate, strong habits, while the LSTM fills in the gaps for broader, context-based patterns (like "it's 9 PM on a Friday, they might open a media app").

⚙️ How Everything Interacts
Data Generation: data_generator.py spits out the user logs.
Training: The run.py script feeds app_usage_logs.csv into predictor.py. The LSTM learns general patterns, and the Markov Chain builds probability tables for each user. The trained weights are saved in the models/ folder.
Simulation & Prediction: simulate.py starts reading through the test data chronologically. Before a user opens an app, the simulator asks the predictor.py engine: "What are the top 3 apps this user is likely to open right now?"
Memory Management (Action): The memory_manager.py takes these probabilities:
Pre-loading: If the ML model is highly confident (e.g., >12% probability) that the user will open the "Gallery" next, the Memory Manager will proactively load it into RAM.
Eviction: If the RAM is full and the user opens a new app, the Memory Manager needs to kick an app out. Instead of just kicking out the oldest app (LRU), it calculates a score: Score = Recency + Frequency + ML Prediction. It kicks out the app with the lowest score, ensuring apps the ML thinks will be used soon stay in memory.
Anti-Thrashing: If an app is evicted but the user opens it right away, the system applies a penalty so it won't be rapidly evicted again.
📊 Data Used for Training
The training data is completely synthetic but designed to mimic real human behavior. It consists of:

Temporal Features: Exact timestamps, converted into cyclic variables (sine/cosine of the hour and day of the week).
Sequential Features: The exact historical sequence of apps opened by a specific user.
User Archetypes: The data generated simulates different types of users (e.g., a "social butterfly" uses Instagram/WhatsApp frequently, while a "professional" uses Email/Calendar frequently) to ensure the ML model can adapt to different personalities.
GenAI Context Workloads: For AI applications, the data includes "inference requests" and "context lengths" to train the system on how to manage large Language Model Key-Value caches effectively without hogging device memory.
